# CLAUDE.md — Project Context for Claude Code

## Project Overview

FinancePro is a personal finance management web application targeting Italian professionals (especially doctors, lawyers, engineers). It features multi-profile support, transaction tracking, budgeting, goals, asset management, and AI-powered classification.

**Tech Stack:**
- Backend: Python 3.12 / FastAPI / SQLAlchemy 2.0 / PostgreSQL with RLS / Pydantic v2
- Frontend: React 19 / TypeScript 5.9 / Vite / TanStack React Query / Tailwind CSS 4
- API Contract: OpenAPI 3.1 auto-generated from Pydantic → Orval generates TS types + React Query hooks
- i18n: Italian (primary) + English via i18next

**Architecture Principle — Single Source of Truth:**
Pydantic models in `backend/app/schemas/` are the SOT. OpenAPI spec is generated from them. Frontend types are generated from OpenAPI via Orval. NEVER create manual TypeScript types that duplicate backend schemas. Always import from `@/api/generated/models`.

## Current State & Known Issues

### ACTIVE BUG CATEGORIES (reported by developer)
The developer reports: TypeScript compilation errors, backend endpoints returning errors, 
and frontend not connecting properly to backend (CORS/auth/URL issues).
Both backend and frontend start but have bugs at runtime.

### Critical Issue #1: Frontend-Backend Auth Flow
The auth flow has a specific mismatch to check:
- Backend `POST /api/v1/auth/login` returns `{"access_token": "...", "token_type": "bearer"}`
- Frontend `AuthContext.tsx` tries BOTH `response.data.accessToken` AND `response.data.access_token`
- This seems handled, but verify the full chain: login → token storage → interceptor → protected endpoints
- The Axios interceptor in `services/api.ts` reads `localStorage.getItem('token')` and sets `Authorization: Bearer {token}`
- Backend `dependencies.py` uses `HTTPBearer()` which expects `Authorization: Bearer {token}` header
- CORS config in `backend/app/config.py` includes `http://localhost:5173` — this should be correct for Vite dev

### Critical Issue #2: TypeScript Compilation Errors
- Run `cd frontend && npm run build` (which runs `tsc -b && vite build`) to get the full error list
- The `tsc -b` step will show ALL type errors
- Common causes in this codebase:
  - Mock API return types don't match Orval-generated types
  - Features migrated halfway (some hooks use mocks, some use Orval)
  - `as any` casts hiding real mismatches
  - Stale generated types (may need `npm run generate:api` first)

### Critical Issue #3: Backend Endpoints Returning Errors
- ALL data endpoints require JWT auth (`Depends(get_current_user)`)
- If frontend sends requests without valid token → 401/403
- If database tables don't exist or migrations aren't applied → 500
- Check: is Alembic migration up to date? Run `cd backend && alembic upgrade head`
- Check: is PostgreSQL running and accessible with correct connection string?

### Critical Issue #4: DB ↔ Backend ↔ Frontend Consistency Mismatches

**VERIFIED MISMATCHES (found by automated analysis):**

1. **FinancialProfile has LEGACY FIELDS that don't exist in DB:**
   - `database_connection_string` and `database_type` exist in BOTH FinancialProfileCreate 
     and FinancialProfileResponse Pydantic schemas, but NOT in the SQLAlchemy model or DB.
   - These are leftovers from the v2.0 distributed-database architecture (abandoned in v2.1).
   - The enum `DatabaseType` in enums.py is also legacy (marked as "Remove after migration").
   - **FIX:** Remove these fields from Pydantic schemas, remove DatabaseType enum, 
     then regenerate OpenAPI + Orval types.

2. **Budget schema has computed fields with no DB backing:**
   - BudgetResponse includes `total_spent`, `remaining`, `usage_percentage`, `category_allocations`
   - These are computed/joined fields, NOT direct DB columns — this is INTENTIONAL and correct
   - BudgetCreate includes `category_allocations` — this is a nested create pattern, also intentional
   - **NO FIX NEEDED** — but the API endpoint must compute these values correctly

3. **FinancialGoal has computed fields:**
   - FinancialGoalResponse includes `is_on_track`, `progress_percentage`  
   - These are computed from contributions vs target — INTENTIONAL
   - **NO FIX NEEDED** — but verify the computation logic in the endpoint

4. **camelCase ↔ snake_case naming mismatch:**
   - Pydantic schemas inherit from CamelCaseModel (in schemas/base.py)
   - This uses `alias_generator=to_camel` + `serialize_by_alias=True`
   - So the OpenAPI spec and JSON responses use camelCase (e.g., `defaultCurrency`)
   - But SQLAlchemy models use snake_case (e.g., `default_currency`)
   - Orval generates TypeScript types in camelCase matching the OpenAPI spec
   - **This is a DESIGN CHOICE, not a bug** — but be aware:
     - When sending data TO the backend, BOTH camelCase and snake_case work 
       (due to `populate_by_name=True`)
     - When receiving data FROM the backend, it's always camelCase
     - The ARCHITECTURE.md says "use snake_case in frontend" — this contradicts the 
       actual Pydantic config which serializes to camelCase. 
       **The frontend should use camelCase to match what the API actually returns.**

5. **migration.sql is UTF-16 encoded:**
   - File `backend/migration.sql` is UTF-16 Little Endian — unusual and may cause issues
   - Alembic migrations in `alembic/versions/` are the authoritative source
   - migration.sql appears to be a standalone SQL dump — verify if it's still needed

## Rules for Code Changes

1. **Never edit files in `frontend/src/api/generated/`** — they are auto-generated
2. **Always use snake_case** for API field names in frontend (matching backend)
3. **Import types only from `@/api/generated/models`** — never create manual TS interfaces for API data
4. **Backend validation is primary** — frontend validation via React Hook Form is secondary/UX-only
5. **All user-facing strings must use i18n** — `t('key')` not hardcoded strings
6. **PostgreSQL UUID columns** — always use `UUID(as_uuid=True)` in SQLAlchemy models
7. **Prefer composition over inheritance** in React components
8. **Test after every change** — run `npm run build` (includes `tsc -b`) to catch TS errors

## Debug Workflow (Recommended Order)

When asked to debug/fix, follow this priority:

### Phase 1: Compilation & Type Safety
- Run `cd frontend && npm run build` and fix ALL TypeScript errors
- Run `cd backend && python -c "from app.main import app"` to verify backend imports
- Remove dead files (.old, unused imports)
- Fix all `as any` casts with proper typing

### Phase 2: Backend Health
- Verify all endpoints respond correctly: `curl http://localhost:8000/docs`
- Run `cd backend && python -m pytest app/tests/ -v` and fix failures
- Check database migrations are up to date

### Phase 3: Frontend-Backend Integration
- For each feature using mock API, check if a real backend endpoint exists
- Replace mock imports with Orval-generated hooks where possible
- Test CRUD operations end-to-end for each feature

### Phase 4: UX & Polish
- Verify all loading states work
- Verify error states display properly (not silent failures)
- Check responsive layout on mobile viewports
- Verify i18n: no hardcoded Italian/English strings outside translation files
